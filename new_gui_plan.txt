# Fitting GUI Rewrite v6 (Operationally Locked)

## Summary
Keep v5 architecture and lock the remaining operational details: explicit revision bookkeeping, strict stale-result policy, deterministic score policy, cache key/invalidation rules, unified fit-result application, and enforceable dependency boundaries.

## Core Decisions (Final)
1. `solver.py` remains unchanged behind `fit_app/fitting/solver_adapter.py`.
2. Qt imports are allowed only in `fit_app/ui/*` and `fit_app/fitting/qt_workers.py`.
3. `SessionController` is plain Python and owns the only mutable `SessionState`.
4. UI receives immutable metadata DTOs only, never mutable internal state or large data arrays.
5. Manual and batch fit results use one DTO and one command handler path.

## Revision and Stale-Result Model
1. Use two revision layers:
   `fit_revision_global` and `target_revision[(file_ref, mapping_id)]`.
2. `fit_revision_global` increments on:
   `ApplyEquationSet`, `UpdateCaptureConfig`, fit-affecting `SessionConfig` changes, function-registry changes.
3. `target_revision[(file_ref,mapping_id)]` increments on:
   `SetParamValue`, `SetBoundaryValue`, and any target-specific lock/unlock affecting fit inputs.
4. `ui_revision` increments only for selection/view commands and is never used for staleness checks.
5. `FitResultDTO` includes:
   `job_id`, `file_ref`, `mapping_id`, `fit_revision_global_at_start`, `target_revision_at_start`.
6. Strict policy:
   if revisions do not match at apply time, reject result as stale (no merge).

## Command Set (Mutation Only)
1. `ApplyEquationSet`
2. `SelectFile`
3. `SelectMapping`
4. `SetParamValue`
5. `SetBoundaryValue`
6. `UpdateCaptureConfig`
7. `ApplyFitResult`

Controller actions (non-command):
1. `start_autofit() -> JobHandle`
2. `start_batch_fit() -> JobHandle`
3. `save_session()`
4. `load_session()`

## Fit Result and Score Policy
1. Persist `FitSnapshotCore` with:
   `score`, `score_name`, `metrics`, `params`, `boundaries`, `updated_at_utc`.
2. v1 canonical score:
   `score_name="r2"`, higher-is-better, non-finite treated as `-inf`.
3. Persist extra metrics in `metrics` (`r2`, `sse`, `rmse` when available).
4. `domain/policies.py` owns `is_improvement(old,new)` with deterministic NaN/tie handling.

## Data Cache and Array Safety
1. Add Qt-free `data/channel_cache.py`.
2. Raw cache key:
   `(file_ref_canonical, file_fingerprint)` where ZIP member key includes archive fingerprint + member path.
3. Derived-transform cache key:
   `(raw_key, data_transform_hash)` from fit-relevant `SessionConfig`.
4. Cached arrays are marked read-only (`writeable=False`).
5. Cache is thread-safe (`Lock`) and provides bounded eviction policy.

## CompiledMapping Lifecycle
1. `CompiledMapping` is never persisted.
2. Controller compile cache key:
   `(mapping_id, equation_set_hash, function_registry_hash)`.
3. Invalidate on:
   `ApplyEquationSet` and function-registry changes.

## Persistence and Portability
1. Session schema stores:
   `source_root_abs`, optional `source_root_rel`, per-file `file_ref_rel`, `file_ref_abs`, fingerprint.
2. Resolve order:
   relative -> absolute -> constrained fingerprint search under source root.
3. Load returns `ResolutionReport`:
   per-file resolution method, unresolved refs, ambiguous matches.
4. Persist errors as:
   `error_code`, `error_context`, `occurred_at_utc`.

## Invariants
1. Runtime local invariants (O(1), command-scoped).
2. Full-session invariants only in tests/debug/explicit validate action.
3. Required local checks:
   `ApplyEquationSet` mapping/param scope validity,
   `SetParamValue` key/type/lock validity,
   `SetBoundaryValue` count/order/domain,
   `ApplyFitResult` job/revision/target validity and policy-compliant best-known updates.

## Dependency and Architecture Enforcement
1. AST-based CI rule:
   no PyQt imports outside UI/Qt-worker modules.
2. AST-based CI rule:
   only `solver_adapter.py` may import `solver`.
3. AST dependency-direction rules:
   `domain` imports stdlib/typing only; core packages cannot import UI/controller incorrectly.

## Test Plan
1. Unit tests for revision increment matrix and stale-result rejection.
2. Unit tests for `is_improvement` score policy (NaN/inf/ties).
3. Controller integration tests (no Qt) for manual and batch flows using shared `ApplyFitResult`.
4. Cache tests:
   key correctness, transform invalidation, read-only safety, ZIP member identity.
5. Persistence tests:
   roundtrip + resolution report behavior for rel/abs/fingerprint paths.
6. CI architecture guard tests for import boundaries and dependency direction.

## Assumptions and Defaults
1. Strict stale-result rejection is used in v1 (no field-level merge).
2. `SetParamValue` and `SetBoundaryValue` always increment target revision.
3. Sidecar default remains `fit_session.json`.
4. New schema only; no legacy import path in v1.
